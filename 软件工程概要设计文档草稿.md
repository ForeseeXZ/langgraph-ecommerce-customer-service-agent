## 1、系统概述

### 1.1、系统简介

本系统是一个面向中文电商售后服务场景的智能客服演示系统，采用前后端分离架构实现。系统以前端对话界面作为交互入口，以后端智能体服务作为核心处理单元，通过大语言模型结合业务工具调用的方式，为用户提供订单查询、库存查询、退款规则说明和售后工单创建等服务。

本系统的前端基于 Next.js 和 assistant-ui 构建，负责完成聊天窗口展示、消息渲染、工具结果可视化和示例问题交互。后端基于 FastAPI 提供统一接口，并使用 LangGraph 构建具备推理与工具调度能力的智能体工作流。系统当前面向课程设计和教学展示，因此业务数据采用本地模拟数据，便于快速部署、结构演示和后续扩展。

从应用场景上看，该系统适用于电商平台售后咨询的原型验证，也可作为智能客服、对话式 AI 应用和 Agent 工具调用机制的教学案例。系统重点体现了自然语言交互、智能体流程编排、模块化设计、数据建模和前后端联动等软件工程设计思想。

为便于后续说明，文中涉及的主要术语定义如下：

1. 智能客服 Agent：指基于大语言模型和工具调用机制构建的自动化服务代理。
2. 工具调用：指智能体根据用户问题自动选择并调用后端业务函数，获取真实业务数据后再组织回答。
3. LangGraph：一种用于构建多节点智能体流程的框架，可描述“模型推理-工具执行-继续推理”的循环过程。
4. assistant-ui：一种面向聊天式应用的前端界面组件库，可快速实现智能对话界面。
5. OpenAI-compatible API：与 OpenAI 聊天接口协议兼容的模型调用接口，可对接多种大模型服务。

### 1.2、系统目标

本系统的总体目标是设计并实现一个可用于教学展示的中文电商售后智能客服原型系统，使系统能够理解用户的自然语言问题，并通过调用后端业务工具返回结构化、可信的售后服务结果。

在功能目标方面，系统应具备以下能力：

1. 支持用户通过自然语言输入售后问题。
2. 支持对订单状态与物流信息进行查询。
3. 支持对商品库存、补货时间等信息进行查询。
4. 支持对退款、退货、换货等规则进行查询与说明。
5. 支持根据订单信息创建售后工单。
6. 支持在前端界面中展示工具调用结果和问答过程。

在性能目标方面，系统应满足以下要求：

1. 在本地开发环境下可稳定运行。
2. 前后端分离明确，模块职责清晰，便于维护和扩展。
3. 聊天响应流程合理，工具调用结果可实时反馈。
4. 对外接口统一规范，便于后期接入数据库或其他业务系统。

在教学目标方面，系统应能体现以下软件工程能力：

1. 完成从需求理解到概要设计的结构化表达。
2. 展示前端、后端、智能体和数据层的协同设计方法。
3. 体现模块划分、接口设计、数据结构设计和扩展性设计思想。

## 2、总体结构设计

### 2.1、软件结构

本系统采用分层式总体结构设计，可划分为表示层、接口层、智能体控制层和数据层四个部分，各层相互协作、相对独立，形成清晰的软件结构。

1. 表示层：由前端聊天界面构成，负责接收用户输入、显示对话内容、展示工具调用结果和示例问题。
2. 接口层：由 FastAPI 提供统一接口，负责前后端消息接收、格式转换和流式返回。
3. 智能体控制层：由 LangGraph 工作流组成，负责调用大模型、判断是否需要调用工具、组织工具执行与回复生成过程。
4. 数据层：由业务工具函数和数据存储组成，负责订单、库存、退款规则和售后工单等业务数据的查询与更新。

系统的调用关系如下：

用户 -> 前端界面 -> FastAPI 聊天接口 -> LangGraph 智能体 -> 业务工具层 -> 数据存储层

其中，系统主要模块可进一步划分如下：

1. 前端页面模块
2. 聊天组件模块
3. 消息渲染模块
4. 工具结果展示模块
5. 聊天接口适配模块
6. 智能体推理模块
7. 工具调度模块
8. 订单查询模块
9. 库存查询模块
10. 退款规则查询模块
11. 售后工单创建模块
12. 数据读写模块

上述模块之间采用低耦合设计。前端仅通过统一接口与后端通信，不直接接触业务数据。智能体层只负责推理与调度，不直接持有底层数据结构。工具层只负责单一业务动作，便于维护和复用。数据层则专注于存储结构组织，便于未来升级为正式数据库系统。

如果需要在文档中配图，可将总体结构图绘制为如下层次：

1. 用户层：用户
2. 界面层：首页、聊天界面、工具结果卡片
3. 服务层：FastAPI 接口服务
4. 智能体层：模型节点、工具节点、流程控制
5. 业务层：订单服务、库存服务、退款规则服务、工单服务
6. 数据层：MySQL 数据库（当前演示版本暂由 JSON 模拟）

### 2.2、设计思想

本系统的设计思想主要体现在以下几个方面。

第一，采用“自然语言交互 + 工具增强”的设计模式。用户无需学习复杂菜单或固定命令，只需输入自然语言问题即可完成售后咨询。对于涉及业务数据的内容，系统优先调用工具获取结构化数据，再据此组织自然语言回答，从而避免大模型直接生成不准确的业务结果。

第二，采用前后端分离设计思想。前端负责交互展示和用户体验，后端负责业务逻辑、流程控制和数据访问，两者通过统一 API 进行通信。该设计有利于团队协作开发，也便于后续前后端独立升级。

第三，采用模块化设计思想。系统将页面展示、接口适配、智能体推理、工具执行和数据管理分离为多个模块，每个模块职责明确，便于后续测试、维护和扩展。

第四，采用可扩展的数据设计思想。尽管当前演示版本使用本地模拟数据完成原型开发，但从概要设计层面已经按关系型数据库进行规划，后续可平滑迁移至 MySQL 数据库，实现更规范的数据约束、索引优化和并发管理。

第五，强调教学演示与工程实践并重。系统既保留了真实业务系统的基本结构，又控制了实现复杂度，适合作为课程实验、答辩展示和功能扩展的基础平台。

## 3、模块设计

### 3.1、模块 1：前端交互模块

#### 3.1.1、功能描述

前端交互模块负责实现用户与系统之间的可视化交互，是整个系统的人机交互入口。该模块主要完成首页展示、聊天线程展示、输入框交互、示例问题快捷发送、用户消息与助手消息的区分展示，以及工具结果卡片化展示等功能。

用户进入系统后，可在聊天输入框中输入订单查询、库存查询、退款咨询和工单创建等问题。系统将输入内容发送到后端接口，并将返回结果以对话形式实时展示。对于工具调用结果，前端模块将其以卡片方式展示关键字段，如订单号、物流状态、库存数量、退款规则摘要和工单编号等，以增强可读性。

#### 3.1.2、接口描述

前端交互模块主要通过 HTTP 接口与后端通信。

1. 接口名称：聊天接口
2. 请求方式：POST
3. 请求地址：`/api/chat`
4. 输入信息：系统提示词、消息列表、可用工具定义
5. 输出信息：流式文本响应、工具调用请求、工具执行结果
6. 异常处理：当接口不可达或响应失败时，前端应提示用户请求失败，并支持重新发送

#### 3.1.3、数据结构描述

前端交互模块涉及的数据结构主要包括：

1. 消息对象：包含角色、消息内容、时间顺序等信息。
2. 用户消息：由用户输入的文本组成。
3. 助手消息：由系统返回的自然语言回答组成。
4. 工具调用结果对象：包含工具名称、状态、消息提示、业务数据载荷等内容。
5. 示例问题列表：用于首页引导用户快速体验系统功能。

#### 3.1.4、实现思路

该模块采用 React 组件化方式开发，并基于 assistant-ui 提供的会话容器、消息组件和输入组件构建聊天界面。通过运行时配置将后端接口地址绑定到聊天组件，当用户发送问题后，由运行时模块将请求提交到后端并接收流式返回。针对不同类型的工具结果，系统配置专门的 UI 渲染逻辑，用于提取关键字段并显示为信息卡片。

#### 3.1.5、其他

该模块属于图形化 Web 应用界面，运行形式为浏览器端页面。界面设计遵循简洁、清晰、便于展示的原则，适合答辩演示和课程作业展示。

### 3.2、模块 2：接口适配模块

#### 3.2.1、功能描述

接口适配模块位于后端服务入口处，负责接收前端发送的聊天请求，并将请求中的消息结构转换为后端智能体能够识别的内部消息格式。同时，该模块还负责将智能体返回的文本、工具调用片段和工具执行结果重新封装为前端可接收的流式数据。

该模块是前后端通信的桥梁，其核心作用是屏蔽前端界面层与智能体内部消息机制之间的差异，保证各层之间数据格式统一、接口清晰。

#### 3.2.2、接口描述

1. 接口名称：聊天完成接口
2. 所属模块：FastAPI 服务接口
3. 输入信息：
   - 系统提示词
   - 工具定义列表
   - 历史消息列表
4. 输出信息：
   - 助手文本内容
   - 工具调用标识与参数片段
   - 工具执行结果
5. 异常处理：
   - 当消息格式非法时，返回参数错误提示
   - 当模型调用异常时，返回服务异常提示
   - 当工具执行异常时，返回工具失败状态

#### 3.2.3、数据结构描述

该模块主要使用以下数据结构：

1. 系统消息结构
2. 用户消息结构
3. 助手消息结构
4. 工具消息结构
5. 工具调用结构
6. 流式响应结构

这些结构用于在前端格式与 LangChain/LangGraph 消息结构之间完成双向转换。

#### 3.2.4、实现思路

在实现上，该模块首先定义统一的请求模型，用于约束前端传入参数。随后根据消息角色，将前端消息转换为系统消息、用户消息、助手消息和工具消息等内部结构。执行过程中，系统采用流式方式接收智能体返回内容，并对文本与工具调用片段分开处理，最终按前端运行时所需格式返回。

#### 3.2.5、其他

该模块以服务程序形式运行，是整个后端对外通信的统一入口。

### 3.3、模块 3：智能体编排模块

#### 3.3.1、功能描述

智能体编排模块是系统的核心模块，负责调用大语言模型理解用户问题，并根据问题内容决定是否调用业务工具。如果模型判断问题涉及订单、库存、退款规则或工单处理，则生成工具调用请求；工具返回结果后，智能体再继续组织最终回复，从而形成完整的问答闭环。

该模块不仅承担自然语言理解任务，还负责将业务规则与工具能力纳入推理过程，使系统具备“可思考、可调用、可回答”的能力。

#### 3.3.2、接口描述

1. 模块名称：智能体工作流模块
2. 输入信息：
   - 系统提示词
   - 历史消息状态
   - 工具定义信息
3. 输出信息：
   - 助手自然语言回复
   - 工具调用请求
   - 下一节点执行方向
4. 异常处理：
   - 当工具不存在时，返回异常中断信息
   - 当模型调用失败时，结束当前流程并返回错误提示

#### 3.3.3、数据结构描述

该模块核心数据结构为消息状态结构 `messages`。该结构用于维护当前会话中的全部消息，包括用户消息、助手消息和工具消息。系统根据消息状态的最新内容判断是否需要继续执行工具节点或结束流程。

#### 3.3.4、实现思路

该模块采用 LangGraph 构建状态图，主要包含两个节点：

1. Agent 节点：负责调用模型，根据消息上下文和系统提示词生成回复或工具调用请求。
2. Tools 节点：负责执行模型选择的具体工具，并将执行结果追加到消息状态中。

系统入口为 Agent 节点。若 Agent 输出中包含工具调用，则流程转入 Tools 节点；工具执行完成后，再返回 Agent 节点继续推理；若 Agent 未产生工具调用，则说明可直接结束流程。

这一结构符合典型的 ReAct 智能体工作模式，能够实现“理解问题-调用工具-整合结果-输出答案”的闭环设计。

#### 3.3.5、其他

该模块以后台逻辑服务形式运行，不直接面向用户界面，但决定了系统整体智能水平与业务处理能力。

### 3.4、模块 4：业务工具模块

#### 3.4.1、功能描述

业务工具模块负责向智能体提供可调用的业务函数，是系统实现可信查询和业务动作执行的关键模块。当前系统已实现以下四项核心工具：

1. 订单状态查询工具：根据订单号查询订单状态、物流信息和预计送达时间。
2. 商品库存查询工具：根据商品编号或商品名称查询库存、仓库和补货时间。
3. 退款规则查询工具：根据问题类型和商品类别查询适用的售后规则。
4. 售后工单创建工具：根据订单号、问题类型、问题描述和联系方式创建工单。

这些工具共同构成售后业务服务能力，为智能体回答提供数据支撑。

#### 3.4.2、接口描述

1. 工具名称：`query_order_status`
   - 输入：订单号、手机号后四位
   - 输出：订单信息、物流信息、状态说明
   - 异常：订单不存在、手机号校验失败

2. 工具名称：`query_product_inventory`
   - 输入：商品编号或商品名称
   - 输出：库存信息、补货时间、仓库信息
   - 异常：商品不存在

3. 工具名称：`query_refund_policy`
   - 输入：问题类型、商品类别
   - 输出：退款规则摘要、适用条件、处理时效
   - 异常：规则不存在

4. 工具名称：`create_after_sales_ticket`
   - 输入：订单号、问题类型、问题描述、联系电话
   - 输出：新工单编号、处理状态、服务承诺
   - 异常：订单不存在、创建失败

各工具统一返回 JSON 风格结果结构，包含状态码、提示信息、业务数据和时间戳。

#### 3.4.3、数据结构描述

业务工具模块主要依赖以下数据结构：

1. 订单对象
2. 商品对象
3. 退款规则对象
4. 工单对象
5. 统一返回结果对象

统一返回结果对象通常包含以下字段：

1. `status`：执行状态
2. `message`：结果说明
3. `data`：业务数据
4. `timestamp`：返回时间

#### 3.4.4、实现思路

该模块将每项业务能力封装为单独函数，并使用统一的工具装饰方式供智能体注册调用。查询类工具通过读取数据源并按条件筛选返回结果，写入类工具则在校验输入正确性后生成业务记录并持久化保存。各工具之间相互独立，便于后续新增更多售后业务能力。

#### 3.4.5、其他

该模块属于后台服务逻辑模块，当前数据来源为本地模拟数据文件。后续可与正式数据库、ERP 系统或订单系统集成。

### 3.5、模块 5：数据管理模块

#### 3.5.1、功能描述

数据管理模块负责管理系统中的业务数据，包括订单数据、商品数据、退款规则数据和售后工单数据。该模块为业务工具提供统一的数据读取与写入能力，是业务层和数据层之间的连接点。

#### 3.5.2、接口描述

1. 输入信息：数据文件路径、查询条件、写入数据对象
2. 输出信息：查询结果集、更新后的数据内容
3. 异常处理：文件不存在、数据格式错误、写入失败等

#### 3.5.3、数据结构描述

当前数据结构采用列表套对象的形式组织，后续数据库实现阶段将转换为标准的表结构与字段结构。

#### 3.5.4、实现思路

当前版本通过读取本地数据文件完成原型验证。在后续升级过程中，该模块将作为数据库访问层进行重构，提供数据访问对象、查询接口和事务支持，以适应正式数据库环境。

#### 3.5.5、其他

该模块属于基础支撑模块，不直接向用户展示，但对业务模块的可用性和可扩展性具有重要作用。

## 4、数据库与数据结构设计

### 4.1、数据库及数据表

从概要设计的角度出发，本系统后续计划采用 MySQL 8.0 作为正式数据库管理系统，以支持结构化数据存储、主外键约束、索引优化、多表关联查询和并发访问控制。当前课程演示版本为了降低部署复杂度，暂以本地 JSON 数据文件模拟数据库内容，但数据库方案已在本节中完成设计，标记为后续进一步实现。

拟设计的数据库名称可设为：`after_sales_agent_db`

数据库主要数据表如下：

1. `customers` 用户表
2. `orders` 订单主表
3. `order_items` 订单明细表
4. `products` 商品表
5. `refund_rules` 退款规则表
6. `after_sales_tickets` 售后工单表
7. `ticket_logs` 工单处理日志表

各数据表的概要设计如下。

1. `customers` 用户表
   - `customer_id`：用户编号，主键
   - `customer_name`：用户姓名
   - `phone`：手机号
   - `member_level`：会员等级
   - `created_at`：创建时间

2. `orders` 订单主表
   - `order_id`：订单编号，主键
   - `customer_id`：用户编号，外键
   - `order_status`：订单状态
   - `after_sales_status`：售后状态
   - `payment_status`：支付状态
   - `payment_method`：支付方式
   - `paid_amount`：实付金额
   - `coupon_amount`：优惠金额
   - `shipping_fee`：运费
   - `order_total`：订单总额
   - `shipping_company`：物流公司
   - `tracking_no`：物流单号
   - `delivery_status`：配送状态
   - `estimated_delivery`：预计送达时间
   - `shipping_address`：收货地址
   - `created_at`：下单时间

3. `order_items` 订单明细表
   - `item_id`：明细编号，主键
   - `order_id`：订单编号，外键
   - `product_id`：商品编号，外键
   - `sku_id`：SKU 编号
   - `product_name`：商品名称
   - `variant_name`：规格信息
   - `unit_price`：单价
   - `quantity`：数量
   - `refund_eligible`：是否支持售后

4. `products` 商品表
   - `product_id`：商品编号，主键
   - `product_name`：商品名称
   - `category`：商品类别
   - `brand`：品牌
   - `price`：价格
   - `status`：销售状态
   - `available_stock`：可用库存
   - `locked_stock`：锁定库存
   - `safety_stock`：安全库存
   - `warehouse`：仓库
   - `restock_eta`：补货时间
   - `specs_json`：规格扩展信息

5. `refund_rules` 退款规则表
   - `rule_id`：规则编号，主键
   - `category`：适用品类
   - `issue_type`：问题类型
   - `rule_summary`：规则摘要
   - `eligible_status`：适用订单状态
   - `conditions_text`：申请条件
   - `required_evidence`：所需凭证
   - `processing_time`：处理时效
   - `refund_scope`：退款范围

6. `after_sales_tickets` 售后工单表
   - `ticket_id`：工单编号，主键
   - `order_id`：订单编号，外键
   - `issue_type`：问题类型
   - `description`：问题描述
   - `contact_phone`：联系电话
   - `status`：工单状态
   - `priority`：优先级
   - `assigned_to`：分配处理组
   - `sla`：服务时效承诺
   - `latest_progress`：最新处理进度
   - `created_at`：创建时间
   - `updated_at`：更新时间

7. `ticket_logs` 工单处理日志表
   - `log_id`：日志编号，主键
   - `ticket_id`：工单编号，外键
   - `operator_name`：处理人
   - `action_type`：处理动作
   - `action_detail`：处理说明
   - `created_at`：记录时间

在后续实现中，可进一步为高频查询字段建立索引，例如 `order_id`、`product_id`、`ticket_id`、`issue_type`、`customer_id` 等，以提高查询效率。

说明：上述数据库结构为系统后续升级目标，当前项目演示版尚未落地 MySQL 实现，现阶段仍采用本地 JSON 模拟数据完成业务功能验证。

### 4.2、数据结构设计

本系统的数据结构主要围绕订单、商品、规则和工单四类核心业务对象展开。

1. 用户数据结构
   - 标识符：`Customer`
   - 数据项：用户编号、姓名、手机号、会员等级、创建时间
   - 层次关系：一个用户可对应多个订单

2. 订单数据结构
   - 标识符：`Order`
   - 数据项：订单编号、用户编号、订单状态、售后状态、支付状态、物流状态、金额信息、收货地址、创建时间
   - 层次关系：一个订单包含多个订单明细

3. 订单明细数据结构
   - 标识符：`OrderItem`
   - 数据项：明细编号、订单编号、商品编号、SKU、商品名称、规格、数量、单价、是否支持售后
   - 层次关系：多个订单明细从属于一个订单

4. 商品数据结构
   - 标识符：`Product`
   - 数据项：商品编号、商品名称、类别、品牌、价格、销售状态、库存信息、仓库、补货时间、规格扩展信息
   - 层次关系：一个商品可出现在多个订单明细中

5. 退款规则数据结构
   - 标识符：`RefundRule`
   - 数据项：规则编号、品类、问题类型、规则摘要、申请条件、所需凭证、处理时效、退款范围
   - 层次关系：一个问题类型可匹配多条业务规则

6. 售后工单数据结构
   - 标识符：`AfterSalesTicket`
   - 数据项：工单编号、订单编号、问题类型、问题描述、联系电话、状态、优先级、分配组、SLA、最新进度、创建时间、更新时间
   - 层次关系：一个订单可关联多个工单

7. 工单日志数据结构
   - 标识符：`TicketLog`
   - 数据项：日志编号、工单编号、处理人、处理动作、处理说明、时间
   - 层次关系：一个工单可对应多条处理日志

8. 工具返回结果数据结构
   - 标识符：`ToolResult`
   - 数据项：状态、消息、数据载荷、时间戳
   - 层次关系：每次工具调用都对应一个结果对象

以上数据结构中，`Customer`、`Order`、`OrderItem`、`Product`、`AfterSalesTicket` 和 `TicketLog` 适合采用关系型数据库表结构实现；而商品规格扩展字段、模型工具结果等可在必要时采用 JSON 类型字段或中间结构进行补充表达。

### 4.3、数据存储设计

本系统的数据存储设计按照 MySQL 关系型数据库方案进行规划，当前尚处于概要设计和原型验证阶段，数据库落地工作待后续进一步实现。

在存储要求方面：

1. 用户、订单、商品、规则和工单等核心业务数据采用结构化字段存储。
2. 对经常查询的订单号、商品编号、工单编号等字段建立索引。
3. 对外键关系进行约束，保证数据一致性和完整性。
4. 对工单更新过程记录处理日志，以支持状态跟踪和责任追踪。

在访问方法方面：

1. 查询类操作采用条件检索方式访问，如按订单号查询订单、按商品编号查询库存、按问题类型查询规则。
2. 写入类操作主要包括工单创建、工单状态更新、日志追加等。
3. 后续可通过 ORM 或数据库访问层统一管理 SQL 操作，提高代码可维护性。

在存取单位方面：

1. 订单数据以订单主表和订单明细表拆分存储。
2. 商品数据按单商品记录存储。
3. 退款规则按规则记录存储。
4. 工单数据按单工单记录存储，并通过日志表维护处理轨迹。

在物理关系方面：

1. `customers` 与 `orders` 之间为一对多关系。
2. `orders` 与 `order_items` 之间为一对多关系。
3. `orders` 与 `after_sales_tickets` 之间为一对多关系。
4. `after_sales_tickets` 与 `ticket_logs` 之间为一对多关系。
5. `products` 与 `order_items` 之间通过商品编号建立关联。

在安全与保密方面：

1. 用户手机号、地址等敏感信息应在正式系统中增加脱敏与权限访问控制。
2. 售后工单数据应限制写入权限，避免误操作。
3. 后续数据库实现中应增加账号认证、角色控制和操作日志记录。

当前说明：现阶段项目源代码仍使用本地 JSON 文件进行数据读取和写入，目的是降低实验复杂度、快速完成功能验证。数据库存储设计已在本节中明确，后续可直接按本方案升级实现。

## 5、接口设计

### 5.1、外部接口

本系统的外部接口主要包括前端访问后端服务接口，以及后端访问大模型服务接口两类。

1. 前端到后端聊天接口
   - 接口类型：HTTP POST
   - 接口地址：`/api/chat`
   - 控制方式：前端发送 JSON 请求，后端以流式方式返回结果
   - 主要输入：
     - 系统提示词
     - 消息列表
     - 工具列表
   - 主要输出：
     - 助手文本回复
     - 工具调用片段
     - 工具执行结果

2. 后端到模型服务接口
   - 接口类型：OpenAI-compatible API
   - 控制方式：通过模型客户端向兼容接口发送对话消息和工具定义
   - 主要输入：
     - 系统提示词
     - 历史消息
     - 可用工具定义
   - 主要输出：
     - 模型生成文本
     - 工具调用请求

3. 后续可扩展外部接口
   - 数据库访问接口
   - 后台管理系统接口
   - 第三方订单系统接口
   - 短信通知或邮件通知接口

### 5.2、内部接口

本系统内部接口主要存在于前端组件、后端服务模块、智能体模块、工具模块和数据模块之间。

1. 前端组件内部接口
   - 页面组件调用聊天组件
   - 聊天组件调用消息渲染组件
   - 工具渲染组件接收工具结果对象并进行展示

2. 后端服务与智能体模块接口
   - 输入：格式转换后的消息列表、系统提示词、工具配置
   - 输出：模型回复或工具调用动作

3. 智能体模块与工具模块接口
   - 输入：工具名称与参数
   - 输出：工具执行结果对象

4. 工具模块与数据模块接口
   - 输入：查询条件或新建工单数据
   - 输出：查询结果或写入结果

5. 数据模块与数据库接口
   - 当前阶段：数据文件读写
   - 后续阶段：数据库访问层调用 MySQL

内部接口设计遵循统一输入、统一输出、职责明确的原则，有利于后续模块替换和功能扩展。
